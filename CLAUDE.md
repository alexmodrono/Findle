# CLAUDE.md

## Purpose

Build and maintain a **native, Finder-native, commercial-quality macOS app** for syncing Moodle / Open LMS course content to the Mac.

This project must feel like a first-class macOS cloud-files product, similar in polish to Dropbox or OneDrive, but specialized for Moodle/Open LMS course content.

The goal is not a quick prototype. The goal is a durable, shippable product architecture.

---

## Non-negotiable product principles

### 1) Native macOS first
This must be a real macOS app built with native Apple frameworks.

Use:
- Swift
- SwiftUI
- AppKit only where needed
- File Provider
- URLSession
- Keychain
- SQLite or similarly robust local persistence
- OSLog
- structured concurrency
- XPC where justified
- SMAppService where justified

Do **not** use:
- Electron
- Tauri
- webview shells
- React Native
- Flutter
- Python as a production runtime dependency
- Node-based desktop wrappers
- external CLI tools as the core runtime architecture

### 2) Finder-native means File Provider
The canonical Finder integration model is **File Provider**, not a plain synced folder.

This means:
- the app should expose content through a File Provider domain
- Finder sidebar presence should be treated as a core feature
- metadata-first sync and placeholders should be first-class
- on-demand materialization is expected
- offline pinning should be supported
- the File Provider domain is the canonical storage surface

A normal user-chosen folder must **not** be the architectural center of the app.

A secondary optional “offline mirror” / “export mirror” feature may exist later, but it must remain secondary.

### 3) Do not build around moodle-dl
Do not build a thin wrapper around `moodle-dl`.

A native Moodle / Open LMS connector must be implemented.

Priority:
1. Official Moodle web services and tokens
2. Moodle mobile-compatible APIs where appropriate
3. An extensible provider abstraction layer for future LMS backends

`moodle-dl` may be used as a reference during development, but it must not be a shipping dependency or runtime requirement.

### 4) Do not build around scraping
Do not make fragile HTML scraping the primary strategy.

If a target LMS instance lacks required APIs:
- detect this explicitly
- surface a clear compatibility limitation
- keep architecture extensible for future adapters
- do not turn scraping into the main design

### 5) Commercial quality over shortcuts
Prefer the most durable, native, maintainable solution.

Do not choose a shortcut that would damage:
- Finder-native behavior
- long-term maintainability
- security/privacy
- code signing / sandbox readiness
- future extensibility
- product polish

Optimize for a **narrow vertical slice on the correct architecture**, not a fast but wrong prototype.

## Repository and Git rules

Git hygiene is a first-class requirement.

### Required workflow
- Initialize a proper Git repository immediately if one does not exist.
- Keep the repository well-structured from the start.
- Make **atomic, logically scoped commits** for every major change.
- Use professional commit messages.
- Keep the working tree clean between major milestones where reasonably possible.
- Do not rewrite history unless explicitly instructed.
- Do not make huge “catch-all” commits.

- `Scaffold app targets and shared domain modules`
- `Implement Moodle token auth client`
- `Add File Provider item model and placeholder enumeration`

Do not use vague messages like:
- `stuff`
- `fixes`
- `work in progress`
- `update`

## Architectural rules

### Canonical storage model
The File Provider domain is the canonical filesystem-facing model.

Do not structure the app around:
- a folder watcher
- a normal folder sync daemon
- directly mirroring everything into `~/Documents/...` as the primary model

### Finder Sync usage
Finder Sync is optional and secondary.

If used:
- it must provide concrete UX value
- it must not hold sync logic
- it must not become the foundation of the product

### Background behavior
Use the appropriate Apple-native mechanisms.

If background components are needed:
- scope responsibilities narrowly
- use `SMAppService` where appropriate
- avoid ad hoc background hacks

### Persistence
Use robust local persistence.

Store:
- server/account metadata
- sync cursors
- item metadata
- local content state
- pinning state
- indexes needed for performance and diagnostics

Do not rely on ephemeral in-memory state for core product correctness.

### Sync engine
The sync engine should support:
- metadata-first enumeration
- incremental refresh
- deterministic conflict handling
- retry/backoff
- cancellation
- invalidation
- per-course and per-item state
- on-demand materialization
- offline pinning
- removal of local materialization where the File Provider model supports it

Avoid hidden state transitions.

Use explicit state modeling.

## Product behavior requirements

The app should support, at minimum:

### Onboarding
- add Moodle/Open LMS server
- validate compatibility
- authenticate cleanly
- explain available access level
- configure the File Provider domain cleanly

### Account management
- secure credentials/tokens in Keychain
- logout
- reconnect
- account switching where reasonable
- clear recovery paths for expired/revoked auth

### Course discovery
- enumerate accessible courses
- allow sync scope selection
- show course metadata clearly

### Content modeling
Represent Moodle content appropriately in Finder.

Support at minimum:
- downloadable files
- folders
- PDFs
- documents
- spreadsheets
- presentations
- images
- archives
- URL-like resources where sensible
- thoughtful treatment of page-like/HTML resources

Do not force poor file abstractions for entities that are not naturally file-like.

### Finder-native behavior
- fast placeholder appearance
- reliable on-demand downloads
- human-friendly filenames
- coherent hierarchy
- sensible status reporting
- compatibility with Quick Look / file opening where applicable

### Diagnostics
Include:
- structured logs
- privacy-safe diagnostics export
- sync timeline or event history
- rebuild index / reset provider / re-authenticate tools where appropriate

## Security and privacy rules

Security and privacy matter.

### Required
- Store secrets in Keychain only.
- Never log secrets.
- Minimize stored sensitive data.
- Treat course content as potentially sensitive.
- Make sync scope user-controllable.
- Prefer privacy-preserving defaults.
- Make telemetry opt-in if telemetry exists.
- Ensure diagnostics can be disabled or limited.

### Avoid
- plaintext credential storage
- broad filesystem assumptions
- excessive entitlements
- storing unnecessary course/user data
- noisy logging of private content

Design for:
- hardened runtime readiness
- code signing
- notarization
- sandbox-aware behavior where feasible

## Code quality rules

### Prefer
- small, well-named types
- explicit state machines
- clear protocols/interfaces where useful
- testable boundaries
- structured concurrency
- OSLog for observability

- isolate it
